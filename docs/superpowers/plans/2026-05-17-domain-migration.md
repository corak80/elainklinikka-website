# Domain Migration: saarivet.fi → elainklinikkasaari.fi

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Make `elainklinikkasaari.fi` (older, better SEO authority) the canonical website, 301-redirect `saarivet.fi` to it, and 301 every renamed old WordPress slug to its new equivalent — while keeping `@saarivet.fi` email working.

**Architecture:**
- GitHub Pages serves the static site, custom domain switched from `saarivet.fi` to `elainklinikkasaari.fi`.
- Two Cloudflare Workers handle redirects:
  - `saarivet-redirect` — every request to `saarivet.fi/*` → 301 to `elainklinikkasaari.fi/<same-path-and-query>`.
  - `elainklinikkasaari-slugs` — requests matching legacy WP slug paths → 301 to the new slug. Other paths pass through to GitHub Pages.
- Both domains' DNS lives on Cloudflare. `saarivet.fi` MX records (Cloudflare Email Routing) stay intact so email keeps working. `elainklinikkasaari.fi` MX records are dropped (no email there).

**Tech Stack:** Cloudflare DNS / Workers / Email Routing, GitHub Pages, plain JavaScript Workers, Bash + curl for verification.

**Estimated total time:** ~3-4 hours of active work spread over 2 days (most "wait time" is DNS propagation and GH Pages cert issuance).

**Key reference data** (gathered 2026-05-17):
- Old WP page count: 96 (page-sitemap) + 12 (post-sitemap) = 108 URLs
- New site URL count: 174 in sitemap
- Old MX: `mail-in.cloudcity.fi` — to be dropped (no email on this domain)
- GH Pages IPs: `185.199.108.153`, `185.199.109.153`, `185.199.110.153`, `185.199.111.153`
- New site CNAME target: `corak80.github.io`
- Existing Cloudflare nameservers (already used by saarivet.fi): `cecelia.ns.cloudflare.com`, `seamus.ns.cloudflare.com`

This Worker redirects every request on `saarivet.fi` (including the apex and every subpath) to the same path on `elainklinikkasaari.fi`. It does NOT proxy — it returns `301 Moved Permanently` so Google transfers link equity.

- [ ] **Step 3.1: Write `redirects/saarivet-worker.js`**

```javascript
// Cloudflare Worker — saarivet.fi → elainklinikkasaari.fi blanket 301
// Worker route: saarivet.fi/*
// Email is NOT touched — Cloudflare Email Routing on saarivet.fi runs at the MX layer, independent of HTTP routing.

const NEW_HOST = "https://elainklinikkasaari.fi";

export default {
  async fetch(request) {
    const url = new URL(request.url);
    const dest = new URL(url.pathname + url.search, NEW_HOST);
    return Response.redirect(dest.toString(), 301);
  },
};
```

## Task 8: Verify every legacy URL redirects correctly

**Files:**
- Create: `redirects/verify.sh`

This script curls every old URL in the slug map and asserts the response is `301` with the correct `Location`.

- [ ] **Step 8.1: Write `redirects/verify.sh`**

```bash
#!/usr/bin/env bash
# Verify every legacy slug 301-redirects to its mapped target.
# Exits non-zero if any URL fails.
set -u

MAP_FILE="$(dirname "$0")/slug_map.json"
BASE="https://elainklinikkasaari.fi"
FAIL=0
TOTAL=0

while IFS=$'\t' read -r old new; do
  TOTAL=$((TOTAL + 1))
  resp="$(curl -sI -o /dev/null -w '%{http_code}|%{redirect_url}' "${BASE}${old}")"
  code="${resp%%|*}"
  location="${resp#*|}"
  expected="${BASE}${new}"
  if [[ "$code" == "301" && "$location" == "$expected" ]]; then
    echo "OK   $old -> $new"
  else
    echo "FAIL $old (got $code -> $location, expected 301 -> $expected)"
    FAIL=$((FAIL + 1))
  fi
done < <(python3 -c "
import json, sys
data = json.load(open('$MAP_FILE'))
for group, rules in data.items():
    if group == '_comment': continue
    for old, new in rules.items():
        print(f'{old}\t{new}')
")

echo
echo "$((TOTAL - FAIL))/$TOTAL redirects working"
exit $FAIL
```

- [ ] **Step 8.2: Make it executable**

Run: `chmod +x ~/elainklinikka-website/redirects/verify.sh`

- [ ] **Step 8.3: Run the verifier**

Run: `~/elainklinikka-website/redirects/verify.sh`

Expected output: every line shows `OK`, final line says `68/68 redirects working`, exit code 0.

If any FAIL: inspect the offending entry. Most failures will be (a) a typo in `slug_map.json` not caught by Task 1's sanity check, or (b) the Worker not picking up the new map (you'd have to update the embedded `SLUG_MAP` in the Worker and redeploy — `slug_map.json` and the Worker are two copies of the same data).

- [ ] **Step 8.4: Spot-check that pass-through (unmapped) URLs return the new site**

Run:

```bash
done
```

Expected output: every line starts with `200`.

## Rollback plan

If anything catastrophic happens in Tasks 5-7 (DNS or Workers misbehave, site goes down):

1. **DNS rollback:** at the registrar, change nameservers back to `ns1.cloudcity.fi`, `ns2.cloudcity.fi`. The old WP site at Webaula resumes serving within minutes-to-hours.
2. **Repo rollback:** the GH Pages site can be restored to `saarivet.fi` by reverting the CNAME commit and pushing:
   ```bash
   cd ~/elainklinikka-website
   git revert <commit-sha-of-CNAME-change>
   git push origin main
   ```
3. **Worker rollback:** disable the `saarivet-redirect` Worker route in Cloudflare; `saarivet.fi` will serve GH Pages directly again.

These rollbacks are why Task 10 (Webaula cancellation) happens last and only after 4+ weeks of verification.

## What this plan does NOT cover

- WordPress media URLs (`/wp-content/uploads/*`) — these will 404 on the new domain. Acceptable: they're not in any sitemap, image hotlinks aren't a primary SEO concern, and embedding a CDN to serve fake replacements isn't worth the complexity. Leave them.
- `/sv/` and `/en/` versions of the `/tietosuoja/` privacy policy — the new site only has the FI version. Both legacy SV/EN privacy URLs redirect to `/tietosuoja/` (FI) per the slug map. Adding SV/EN privacy translations is a separate task.
- Adding `elainklinikkasaari.fi` to other tools (Cardiac Report's allowed origins, Provet integrations, etc.) — those don't reference the website domain, so no changes expected. If something breaks post-cutover, treat as a follow-up.
